Design Considerations

Openness(对开发人员)

Interfaces to the OS, network and services must be based on published specifications (such as standards)

  • This makes it possible for one vendor to build systems which interact with those of another
  • Without such openness, the user is in practice restricted to purchase from only one vendor
  • Different users will buy from different vendors, but cannot later build a distributed system

Too many standards can be almost as bad as vendor proprietary standards, since the user must then deal with all the different standards.

Concurrency

Multiple clients may try to access a service.

The whole system is multi-process, so issues related to multi-process programming should be considered:

  • lock
  • Synchronizing

Scalability(伸缩性) and state

Should avoid:

  • the notion of single master services
  • short fixed names(对一些参数的命名，不能太短)
  • performance bottlenecks

Some resources include some sort of state information: data that alters the resources’ future response to requests, e.g.

  • The state of a file server is the data held on disk
  • Alterations to the state alter the response given to a READ request

Duplication of state information leads to the problem of propagating changes of state to the various copies

可以理解为数据的一致性consistency

Fault tolerance

  • Fault isolation: fault occurring in one resource does not effect operation of other resources
  • Fault masking: take action to restore the service when a particular resource of the service fails

SO hardware redundancy and software recovery are required

Measure of failure rate

Mean Time before Failure(MTBF)

  • Average period of time between failures
  • Assume a random distribution of failures
  • Can compute the probability of failure occurring in any period of time

T-fault tolerance

Must be more than t failed components before a service fails

Failure modes and illustrations

  • Failstop:
    • a processor halts, and remain halted. The fact that the processor has halted can be detected by other processors
  • Crash:
    • As for failstop, except not detectable by other processors
  • Crash+link:
    • A link fails for a period, losing some messages(信息传递通道的fail)
  • Receive omission:
    • Processor fail to read message sent to it
  • Send omission:
    • processor fail to write message required by protocols
  • General ommision:
    • receive or send omission
  • Byzantine failure:
    • the processor behaves in a manner not permitted by the protocol

Transparency(对用户)

Transparency is about making the service provided independent from how it is provided.

Types of transparency:

  • Access transparency:
    • Methods to access a resource do not vary according to means of access, e.g. Java method call, SOAP message
  • Location Transparency:
    • Access to resource does not require specifying its location.
  • Concurrency Transparency:
    • Access to a resource is not affected by number of clients
  • Replication Transparency:
    • Methods to access a resource do not change according to how many copies of that resource are maintained
  • Failure Transparency:
    • A service remains available even if a resource which supplies that service fails Migration Transparency: Moving a resource between hosts does not affect access to it
  • Performance Transparency:
    • Increases or decreases in resource performance do not alter methods by which the resource is accessed
  • Scaling Transparency:
    • Access to resources does not change according to overall size of the system or number of instances of resource

Security

  • Preventing users doing things they should not do, according to policy decisions

  • Preventing actions which by accident prevent the correct functioning of the system

Management

Management is the supervising and controlling of the system so that it satisfies the requirements of its owners and users.

Requirements:

  • Remote access
  • Programmatic interface
  • Relevance:
    • only present necessary information
  • Uniformity

Performance

  • Responsiveness
  • Throughput
  • Load Balancing

Heterogeneity(异质性)

Communication Models

  • Client-server communication
  • Multicast communication

Reliable agreement

Many improtant functions such as data duplications, requre agreement between hosts. This is often achieved through broadcast of information among distributed processes.

Communication links are not reliable. Therefore, distributed algorithms allow us to achieve reliable broadcasts

Data Distributed models

  • Fragmentation:分片
    • data is divided into parts and split between hosts
  • Migration:迁移
    • data moves from one host to another over time

Distributed Transcation(分布式事务)

Ensuring that either all or none of a sequence of actions are performed maintains the integrity of the resources acted on.确保无论执不执行操作，都会保持资源的完整性。

Distributed File System

Provide access to directories and files spread acorss multiple hosts as if one directory structure.
