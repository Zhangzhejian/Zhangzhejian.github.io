TCP报文段

建立连接：三次握手

详解LINK

  • 过程
  • 为什么要三次握手
    • 问题的本质是，信道是不可靠的，但是我们要建立可靠的连接发送可靠的数据，也就是数据传输是需要可靠的。在这个时候三次握手是一个理论上的最小值，并不是说是tcp协议要求的，而是为了满足在不可靠的信道上传输可靠的数据所要求的。ref

  • 如果出现包丢失：

    1. 如果第一个包，A发送给B请求建立连接的报文(SYN)如果丢掉了，A会周期性的超时重传，直到B发出确认(SYN+ACK)；

    2. 如果第二个包，B发送给A的确认报文(SYN+ACK)如果丢掉了，B会周期性的超时重传，直到A发出确认(ACK)；

    3. 如果第三个包，A发送给B的确认报文(ACK)如果丢掉了
      • A在发送完确认报文之后，单方面会进入ESTABLISHED的状态，B还是SYN_RCVD状态
      • 如果此时双方都没有数据需要发送，B会周期性的超时发送(SYN+ACK)，直到收到A的确认报文(ACK)，此时B也进入ESTABLISHED状态，双方可以发送数据；
      • 如果A有数据发送，A发送的是(ACK+DATA)，B会在收到这个数据包的时候自动切换到ESTABLISHED状态，并接受数据(DATA)；
      • 如果这个时候B要发送数据，B是发送不了数据的，会周期性的超时重传(SYN+ACK)直到收到A的确认(ACK)B才能发送数据。

断开连接：四次挥手

  • 过程
  • 为什么要四次挥手

本质的原因是tcp是全双公的，要实现可靠的连接关闭，A发出结束报文FIN，收到B确认后A知道自己没有数据需要发送了，B知道A不再发送数据了，自己也不会接收数据了，但是此时A还是可以接收数据，B也可以发送数据；当B发出FIN报文的时候此时两边才会真正的断开连接，读写分开

流量控制：滑动窗口协议

  • 是什么

  • TCP传输效率问题

    每个报文包中数据过小，导致TCP首部（21字节）+IP首部（21字节）占大部分，使得效率降低。

    ref:https://www.cnblogs.com/zhaoyl/archive/2012/09/20/2695799.html

    • 发送端：

      发送端每次发送端数据过小。

      • nagle算法

        缓存发送端数据。

        1. 先发送第一个字节
        2. 缓存剩下的数据
        3. 等待一个RTT，或者ACK
        4. 发送剩下数据

    • 接收端：糊涂窗口综合征

      接收方读取过慢，导致窗口大小过小，使得每次传输数据太小，导致效率问题

      • clark算法

        只要有数据到达就发送ACK，但是宣布窗口大小为0，直到缓存空间能放下MSS(最大报文段长度)或者一半缓存空间为空

      • 延迟ACK

        收到报文段不立即发送ACK，直到缓存有足够空间。

拥塞控制

发送发维持cwnd的大小，通过cwnd的大小控制发送窗口的大小。

REF：https://juejin.im/post/5acae2b55188252b0b2013aa

拥堵发生前的算法

  • 慢开始算法(slow start)
    1. 连接建立，cwnd=1
    2. 每收到一个ACK，cwnd+=1.
    3. 每经过一个RTT,cwnd=cwnd*2
    4. 当cwnd>=sshthresh(slow start threshold)， 进入拥堵避免算法
  • 拥塞避免
    1. 每收到一个ACK，cwnd+= 1/cwnd
    2. 每经过一个RTT, cwnd+=1

拥堵发生时的算法

拥塞发生时即出现丢包现象。

  1. 超时重传RTO。即消息发送后经过一个RTO也没收到任何ACK。认为网络情况比较糟糕
    1. Ssthresh = cwnd/2
    2. cwnd =1 进入慢开始过程
  2. 快重传。收到重复的3个ACK。认为网络情况比较i好，只是某个包传输失败。
    1. ssthresh = cwnd/2
    2. cwnd = cwnd/2 进入快恢复算法
  • 快重传
  • 快恢复
    1. 收到重复3个ACK，cwnd = ssthresh+3
    2. 发送对应数据
    3. 如果继续收到重复ACK，即传输又失败，cwnd+1，再发送一遍
    4. 如果收到新的ACK即传输成功，cwnd = ssthresh，进入拥塞避免算法
