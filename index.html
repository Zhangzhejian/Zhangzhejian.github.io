Design Pattern

Posted on 2019-12-18

六大原则(SOLID)

  • 单一职责原则Single Responsibility Principle
    • 一个类应该只做一件事
    • 功能的分离
  • 开闭原则 Open-Closed Principle
    • 对拓展开放，对修改关闭
    • 通过接口实现
  • 里氏替换原则Liskov Substitution Principle
    • 子类不应该重写或者重载父类的方法
    • 如果对象 A 是对象 B 的子类，或者对象 A 实现了接口 B（本质上讲，A 就是 B 的一个实例），那么我们应该能够在不做任何特殊处理的情况下，像使用一个对象 B 或者 B 的一个实例那样使用对象 A。
  • 接口隔离原则Interface Segregation Principle
    • 应该保持接口短小，在实现时选择实现多个小接口而不是庞大的单个接口
  • 依赖倒置原则Dependency Inversion Principle
    • 高级模块不应该依赖具象的低级模块，它们都应该依赖相应模块的抽象层
    • 抽象不应该依赖细节；细节应该依赖抽象
  • 迪米特法则
    • 一个对象应该对其他对象保持最少的了解。
    • 类与类之间的关系越密切，耦合度越大，当一个类发生改变时，对另一个类的影响也越大

TCP报文段

建立连接：三次握手

详解LINK

  • 过程
  • 为什么要三次握手
    • 问题的本质是，信道是不可靠的，但是我们要建立可靠的连接发送可靠的数据，也就是数据传输是需要可靠的。在这个时候三次握手是一个理论上的最小值，并不是说是tcp协议要求的，而是为了满足在不可靠的信道上传输可靠的数据所要求的。ref

  • 如果出现包丢失：

    1. 如果第一个包，A发送给B请求建立连接的报文(SYN)如果丢掉了，A会周期性的超时重传，直到B发出确认(SYN+ACK)；

    2. 如果第二个包，B发送给A的确认报文(SYN+ACK)如果丢掉了，B会周期性的超时重传，直到A发出确认(ACK)；

    3. 如果第三个包，A发送给B的确认报文(ACK)如果丢掉了
      • A在发送完确认报文之后，单方面会进入ESTABLISHED的状态，B还是SYN_RCVD状态
      • 如果此时双方都没有数据需要发送，B会周期性的超时发送(SYN+ACK)，直到收到A的确认报文(ACK)，此时B也进入ESTABLISHED状态，双方可以发送数据；
      • 如果A有数据发送，A发送的是(ACK+DATA)，B会在收到这个数据包的时候自动切换到ESTABLISHED状态，并接受数据(DATA)；
      • 如果这个时候B要发送数据，B是发送不了数据的，会周期性的超时重传(SYN+ACK)直到收到A的确认(ACK)B才能发送数据。

断开连接：四次挥手

  • 过程
  • 为什么要四次挥手

本质的原因是tcp是全双公的，要实现可靠的连接关闭，A发出结束报文FIN，收到B确认后A知道自己没有数据需要发送了，B知道A不再发送数据了，自己也不会接收数据了，但是此时A还是可以接收数据，B也可以发送数据；当B发出FIN报文的时候此时两边才会真正的断开连接，读写分开

流量控制：滑动窗口协议

  • 是什么

  • TCP传输效率问题

    每个报文包中数据过小，导致TCP首部（21字节）+IP首部（21字节）占大部分，使得效率降低。

    ref:https://www.cnblogs.com/zhaoyl/archive/2012/09/20/2695799.html

    • 发送端：

      发送端每次发送端数据过小。

      • nagle算法

        缓存发送端数据。

        1. 先发送第一个字节
        2. 缓存剩下的数据
        3. 等待一个RTT，或者ACK
        4. 发送剩下数据

    • 接收端：糊涂窗口综合征

      接收方读取过慢，导致窗口大小过小，使得每次传输数据太小，导致效率问题

      • clark算法

        只要有数据到达就发送ACK，但是宣布窗口大小为0，直到缓存空间能放下MSS(最大报文段长度)或者一半缓存空间为空

      • 延迟ACK

        收到报文段不立即发送ACK，直到缓存有足够空间。

拥塞控制

发送发维持cwnd的大小，通过cwnd的大小控制发送窗口的大小。

REF：https://juejin.im/post/5acae2b55188252b0b2013aa

拥堵发生前的算法

  • 慢开始算法(slow start)
    1. 连接建立，cwnd=1
    2. 每收到一个ACK，cwnd+=1.
    3. 每经过一个RTT,cwnd=cwnd*2
    4. 当cwnd>=sshthresh(slow start threshold)， 进入拥堵避免算法
  • 拥塞避免
    1. 每收到一个ACK，cwnd+= 1/cwnd
    2. 每经过一个RTT, cwnd+=1

拥堵发生时的算法

拥塞发生时即出现丢包现象。

  1. 超时重传RTO。即消息发送后经过一个RTO也没收到任何ACK。认为网络情况比较糟糕
    1. Ssthresh = cwnd/2
    2. cwnd =1 进入慢开始过程
  2. 快重传。收到重复的3个ACK。认为网络情况比较i好，只是某个包传输失败。
    1. ssthresh = cwnd/2
    2. cwnd = cwnd/2 进入快恢复算法
  • 快重传
  • 快恢复
    1. 收到重复3个ACK，cwnd = ssthresh+3
    2. 发送对应数据
    3. 如果继续收到重复ACK，即传输又失败，cwnd+1，再发送一遍
    4. 如果收到新的ACK即传输成功，cwnd = ssthresh，进入拥塞避免算法

接口Interface

继承只能是单继承，接口的实现可以实现多个接口。（弥补java单继承的缺陷）

特点：

  1. 接口中所有属性的默认修饰符是 public static final 不可修改切定义时必须赋值
  2. 接口中速游方法的默认修饰符是 public abstract
  3. 实现接口时，必须实现接口的所有方法。
  4. 接口没有构造方法，无法被实例化
  5. 访问接口中的属性不能用super关键字，直接通过接口名访问
  6. 接口可以多继承接口

Java 多态

多态（英语：polymorphism），是指计算机程序运行时，相同的消息可能会送给多个不同的类别之对象，而系统可依据对象所属类别，引发对应类别的方法，而有不同的行为。简单来说，所谓多态意指相同的消息给予不同的对象会引发不同的动作称之。

多态存在的条件：

  1. 继承关系(或者接口的实现)

  2. 重写：子类对父类的某些方法进行了重写，在调用这些方法的时候就会调用子类的方法

  3. 向上转型：将子类的引用赋给父类对象

多态的特点

  1. 只能重写父类的方法，不能重写父类的成员变量
  2. 方法调用发生时被引用对象的类型决定了哪个方法被调用。

线程池

  • newCachedThreadPool创建一个可缓存线程池，如果线程池长度超过处理需要，可灵活回收空闲线程，若无可回收，则新建线程。
  • newFixedThreadPool 创建一个定长线程池，可控制线程最大并发数，超出的线程会在队列中等待。
  • newScheduledThreadPool 创建一个定长线程池，支持定时及周期性任务执行。
  • newSingleThreadExecutor 创建一个单线程化的线程池，它只会用唯一的工作线程来执行任务，保证所有任务按照指定顺序(FIFO, LIFO, 优先级)执行。

线程安全

  • 内存可见
    • 控制的是线程执行结果在内存中对其它线程的可见性
  • 执行控制
    • 目的是控制代码执行（顺序）及是否可以并发执行

Volatile和synchronized的区别

  1. volatile本质是在告诉jvm当前变量在寄存器（工作内存）中的值是不确定的，需要从主存中读取； synchronized则是锁定当前变量，只有当前线程可以访问该变量，其他线程被阻塞住。
  2. volatile仅能使用在变量级别；synchronized则可以使用在变量、方法、和类级别的
  3. volatile仅能实现变量的修改可见性，不能保证原子性；而synchronized则可以保证变量的修改可见性和原子性
  4. volatile不会造成线程的阻塞；synchronized可能会造成线程的阻塞。
  5. volatile标记的变量不会被编译器优化；synchronized标记的变量可以被编译器优化

EIS(Enterprise information system)

  • An enterprise information system (EIS) is a software system which holds core business data for a company or organisation and which performs operations involving this data.
  • usually large and complex systems(Distributed)
  • be used by different applications as a resource

EIS architecture and components

  1. Client tier
    • responsible to display information and receive information
    • HTML
  2. Presentation tier
    • Managing presentation of information
    • controlling interaction sequence
    • Relaying user requests
    • JSP
  3. Business tier
    • Components implementing the business rules and data
    • Beans
  4. Integration tier
    • Mediates between the business and resource tiers.
    • Manages data retrieval and data conversion.
    • Insulates higher tiers from direct knowledge of low-level data concepts.
    • DB Interfaces
  5. Resource tier
    • Contains persistent data storage and external services.
    • DB

EIS Business tier design

Entity beans vs session beans

  • Session bean
    • Dedicated to a single client
    • Live only for the duration of the client’s session
    • Not persistent
    • Used to model stateful or stateless interactions between the client and business tier components.
    • A session bean encapsulates a group of operations, usually corresponding to the use cases of the system, which operate on one or more of the entities of the system.
    • Each use case of the system would normally be implemented by a session bean
    • 一个对一个client，生存周期为会话的周期，非持久，用来描述client和应用层组件的交互，每个用例通常都可以用一个session bean表示。
  • Entity bean
    • Provides an object view of persistent data
    • Coarse-grained
    • Multi-user and Long-lived
    • Entity beans act as an object-oriented facade for data of system, which may actually be stored as relational tables, or as XML datasets
    • Normally each entity from the PIM class diagram of the system will be managed by some entity bean.
    • 持久的数据，粗粒度，多用户长生存周期。entity bean是系统数据的对象化表示，PIM类图中的每个实体都可用其表示

Stateless session bean vs Stateful session bean

  • Stateless
    • Provide a service by a single method call
    • Instantaneous
    • can be shared and reused
  • Stateful
    • Provide a service specific to a single client
    • cannot be shared
    • Involve several related operations

EIS design issues

  • Data security
    • Password encryption
    • HTTPS
  • Separating presentation, business logic and data processing code
    • EIS components: session beans for business processing, entity beans for complex bbusiness data
    • Separate tiers
  • Pooling database connections
    • Database connection pool
