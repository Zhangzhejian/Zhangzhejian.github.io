Design Considerations

Openness(对开发人员)

Interfaces to the OS, network and services must be based on published specifications (such as standards)

  • This makes it possible for one vendor to build systems which interact with those of another
  • Without such openness, the user is in practice restricted to purchase from only one vendor
  • Different users will buy from different vendors, but cannot later build a distributed system

Too many standards can be almost as bad as vendor proprietary standards, since the user must then deal with all the different standards.

Concurrency

Multiple clients may try to access a service.

The whole system is multi-process, so issues related to multi-process programming should be considered:

  • lock
  • Synchronizing

Scalability(伸缩性) and state

Should avoid:

  • the notion of single master services
  • short fixed names(对一些参数的命名，不能太短)
  • performance bottlenecks

Some resources include some sort of state information: data that alters the resources’ future response to requests, e.g.

  • The state of a file server is the data held on disk
  • Alterations to the state alter the response given to a READ request

Duplication of state information leads to the problem of propagating changes of state to the various copies

可以理解为数据的一致性consistency

Fault tolerance

  • Fault isolation: fault occurring in one resource does not effect operation of other resources
  • Fault masking: take action to restore the service when a particular resource of the service fails

SO hardware redundancy and software recovery are required

Measure of failure rate

Mean Time before Failure(MTBF)

  • Average period of time between failures
  • Assume a random distribution of failures
  • Can compute the probability of failure occurring in any period of time

T-fault tolerance

Must be more than t failed components before a service fails

Failure modes and illustrations

  • Failstop:
    • a processor halts, and remain halted. The fact that the processor has halted can be detected by other processors
  • Crash:
    • As for failstop, except not detectable by other processors
  • Crash+link:
    • A link fails for a period, losing some messages(信息传递通道的fail)
  • Receive omission:
    • Processor fail to read message sent to it
  • Send omission:
    • processor fail to write message required by protocols
  • General ommision:
    • receive or send omission
  • Byzantine failure:
    • the processor behaves in a manner not permitted by the protocol

Transparency(对用户)

Transparency is about making the service provided independent from how it is provided.

Types of transparency:

  • Access transparency:
    • Methods to access a resource do not vary according to means of access, e.g. Java method call, SOAP message
  • Location Transparency:
    • Access to resource does not require specifying its location.
  • Concurrency Transparency:
    • Access to a resource is not affected by number of clients
  • Replication Transparency:
    • Methods to access a resource do not change according to how many copies of that resource are maintained
  • Failure Transparency:
    • A service remains available even if a resource which supplies that service fails Migration Transparency: Moving a resource between hosts does not affect access to it
  • Performance Transparency:
    • Increases or decreases in resource performance do not alter methods by which the resource is accessed
  • Scaling Transparency:
    • Access to resources does not change according to overall size of the system or number of instances of resource

Security

  • Preventing users doing things they should not do, according to policy decisions

  • Preventing actions which by accident prevent the correct functioning of the system

Management

Management is the supervising and controlling of the system so that it satisfies the requirements of its owners and users.

Requirements:

  • Remote access
  • Programmatic interface
  • Relevance:
    • only present necessary information
  • Uniformity

Performance

Reliable agreement

Many improtant functions such as data duplications, requre agreement between hosts. This is often achieved through broadcast of information among distributed processes.

Communication links are not reliable. Therefore, distributed algorithms allow us to achieve reliable broadcasts

Data Distributed models

  • Fragmentation:分片
    • data is divided into parts and split between hosts
  • Migration:迁移
    • data moves from one host to another over time

Distributed Transcation(分布式事务)

Ensuring that either all or none of a sequence of actions are performed maintains the integrity of the resources acted on.确保无论执不执行操作，都会保持资源的完整性。

Lecture 9: Argumentation

Key points:

  • Introduction

  • Constructing arguments

Introduction

What’s Argument

What’s Argumentation

Constructing Argument

Argument dialogues

What’s Argument Dialogues

auctions and negotiation, argument dialogues are a mechanism that agents can use to reach agreement.

By exchanging arguments, agents are able to

  • share their reasons for holding particular positions

  • and perhaps change each other’s position on a topic

Influential argument dialogue typology

Elements of a dialogue system

  • Moves that the agents can make (e.g. assert, accept, claim, question, challenge …)
  • Protocol – rules that state what moves it is permissible to make at any point in the dialogue (e.g. you can’t assert an argument that attacks something you’ve asserted earlier, if you can’t attack an argument then you must accept it …)
  • Strategy – private to an agent, determines which of the permissible moves to make (e.g. if you find something agreeable then agree to it, if you have a choice between an assert and accept move then make the assert move …)
  • Rules that state when the dialogue terminates and what the outcome of the dialogue is (e.g. dialogue terminates when each agent runs out of moves, agent x persuades agent y if agent x was the last agent to move …)

Lec 8: Auctions

Key points:

  • English auctions
  • dutch auctions
  • first price sealed bid auctions
  • vickrey auctions
  • combinatorial auctions
    • Vickrey-clarke-groves mechanism

Introduction

Auctions are used to allocate scarce resources that are desired by more than one agent

Auctioneer aims to allocate a good to one of the bidders.

Auctioneer wants to maximise the selling price. Attempts to do this through the design of an appropriate auction.

Bidders want to minimise the price at which the good is auctioned. Attempt to do this by using an appropriate strategy.

  • Public/common value
  • Private value
  • correlated value
  • open cry
  • sealed bid
  • ascending
  • descending
  • one shot
  • first price
  • Second price: highest wins the good and pays the second highest number

English Auction

  • For allocation of a single item.

  • First price, open cry, ascending.

Dutch auction

  • For allocation of a single item.

  • First price, open cry, descending.

First-price sealed bid auctions

  • For allocation of a single item.

  • First price, sealed bit, one shot.

Vickrey Auction

  • For allocation of a single item.

  • Second price, sealed bit, one shot.

Collusions(合谋)

all the bidders can form a coalition (the grand coalition) and agree to buy the item for a low price from the auctioneer. The coalition can then sell the item for a higher price and split the profit between themselves.

Lecture 7: Social Choice Theory

Introduction

Social choice theory is about how to make group decisions.

Specifically, it provides a range of voting protocols that can be used to account for the preferences of a group of agents.

Fundamental problem of social choice theory is one of preference aggregation:

given a collection of preference orders, one for each voter, how do we combine these to derive a group decision, that reflects as closely as possible the preferences of the voters?

  • Social welfare functions

    • produces a social preference order

  • Social choice functions

    • selects a single outcome

Pluralit Voting procedure

Each candidate gets one point for every preference order that ranks them first.

作为首选项的人数多的获胜

condorcet winner

Condorcet winner is an outcome that beats every other outcome in pairwise majority contests.

逐对比较，每组都获胜的，最后获胜

Condorcet paradox

For every possible candidate, there is another candidate that is preferred by the majority of the voters!

This is Condorcet’s paradox. There are situations in which, no matter which outcome we choose, a majority of voters will prefer another candidate.

因为偏好的传递性导致的悖论

Linear sequential majority elections
Instant runoff rule

Series of mini-elections in which the candidate with the fewest votes in each round is eliminated.

Each round:

• votes are counted for everyone’s first choice and the one with the fewest votes is eliminated;

• any voter who had the eliminated option as their first choice, the votes for their second choice are now counted and we go again.

The last candidate remaining is the winner.

Copeland rule

All candidates face pairwise majority contest against all other candidates.

Each candidate receives 1 point for every pairwise majority contest that they win…

…and loses 1 point for every pairwise majority contest that they lose.

Borda count

Each candidate receives (𝑚 − 1) points for each voter that lists them as their first choice, (𝑚 − 2) points for each voter that lists them as their second choice etc (where 𝑚 is the number of candidates).

spoiler effect: outcome can depend on the presence of alternatives that intuitively seem irrelevant. “Spoilers” shift the outcome from one highly ranked candidate to another.

Positional scoring rules(PSR)
  • Anonymity
    • A voting rule is said to be anonymous if all voters are treated symmetrically.
  • Resolute
    • if there is always a unique winner (no ties).
  • Surjective
    • for every candidate there is some profile of voter preferences such that the candidate will win.
  • Dictatorship
    • if there is some voter 𝑖 such that the winner is the first choice of 𝑖.(The dictator always picks the winner.)
  • Strategy-proof
Weakly Pareto

A voting rule is said to be weakly Pareto if it does not elect some candidate 𝜔 whenever there is some other candidate 𝜔 𝑗 such that all voters prefer 𝜔 𝑗 to 𝜔 𝑖 .

Independence of irrelevant alternatives

(Neither do plurality, instant runoff or sequential majority elections satisfy IIA)

DNS

  • A distributed database implemented in hierarchy of many name servers
  • An application layer protocol that allows hosts to communicate with name servers to resolve names (address/name translation)
Three classes of DNS server
  • Root DNS Server
    • 13 such root DNS servers
  • Top-level domain servers
    • top-level domain servers for example .com
  • Authoritative DNS Servers
    • contains authoritative DNS servers which provide actual answers to the DNS queries.

DNS resource record (RR)

In DNS servers all information including hostnames to IP translation is kept in the so-called Resource Records (RR). A Resource Record is a 4-tuple containing this info: {Name, Value, Type, TTL}.

  • A
    • Map a hostname to a ip
  • NS
    • Which Name Server is authoritative for a domain
  • CNAME
    • used to make one domain an alias(别名) of another
  • MX
    • a list of mail exchange server

DNS Message

DNS Threat

DNS poisoning

With cache poisoning an attacker attempts to insert a fake address record for an Internet domain into the DNS

If the server accepts the fake record, the cache is poisoned and subsequent requests for the address of the domain are answered with the address of a server controlled by the attacker.

(WI) Web Server and HTTP

Posted on 2019-01-04

HTTP

Designed for communicating enriched documents between computers

Proxy

A proxy acts on behalf of other clients and presents requests from other clients to a server. The proxy acts as a server in interacting with a client and as a client in interacting with a server.

  • client and server be separated by security intermediary (firewall).
  • Different Version of HTTP.
Gate way

A gateway is a server that appears to the client as if it were an origin server. It acts on behalf of other servers that may not be able to communicate directly with a client

  1. Client talks to gateway host as if final server
  2. Gateway passes data on to final server
  • Security intermediary:The client and server may be separated by a security intermediary such as a firewall, with the gateway on the server side of the firewall.
  • Non-HTTP server: Web browsers have built into them the capability to contact servers for protocols other than HTTP, such as FTP servers
Tunnel

Client talks to server as if tunnel was not there. Tunnel does nothing but forwards messages

Tunnels are used when there must be an intermediary system between client and server but it is not necessary for that system to understand the contents of messages. Eg. An example is a firewall in which a client or server external to a protected network can establish an authenticated connection.
