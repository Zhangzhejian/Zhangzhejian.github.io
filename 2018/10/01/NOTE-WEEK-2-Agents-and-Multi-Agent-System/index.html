Objects and Agents

Main differences

  • agents are autonomous

They decide for themselves whether or not to perform an action

  • agents are smart

capable of flexible(reactive, pro-active, social) behaviours

  • agents are active

continuous sense-decide-act loop, not passive service providers

Agent behaviour

Agent is a function which maps runs to actions

An agent makes a decision about what action to perform based on the history of the system that it has witnessed to date. Let $\mathcal{AG}$ be the set of all agents. (Agent做的就是根据输入的环境状态，给出决定的动作)

A system is a pair (Ag, Env) containing an agent and an environment. Any system will have associated with it a set of possible runs; we denote the set of runs of agent Ag in environment Env by $\mathcal{R}(Ag, Env)$ (We assume $\mathcal{R}(Ag, Env)$ contains only terminated runs) ($\mathcal{R}(Ag, Env)$是在Env环境下，Ag做出的所有run的集合，假设只包含所有完整的run)

Purely Reactive Agents

Some agents decide what to do without reference to their history — they base their decision making entirely on the present, with no reference at all to the past

Agents with state

Perception subsystem

The perception function see maps an environment state to a percept:

The action-selection function action maps an internal state to an action:

Agents with state

Internal data structure state typically used to record information about the environment state and history. $\mathcal{I}$ be the set of all internal states of the agent

The function next is introduced, which maps an internal state and a percept to an internal state, and then the action-selection function action maps an internal state to an action.

Control Loop
  1. Agent starts in some initial internal state $i_0$
  2. Observes its environment state e, and generates a percept $see(e)$
  3. Internal state of the agent is then updated via next function, becoming $next(i_0 , see(e))$
  4. The action selected by the agent is $action (next(i_0 , see(e)))$
  5. Goto 2

Tasks for Agents

How to specify the tasks for agents is a problem

Maxmise utility
Associate utilities with individual states

the task specification is a function which asscociates a real number with every environments state

Disadvantage: difficult to specify a long term view when assigning utilities to individual states

Associate utilities with runs

Example: trading agent might use $u(r) = (p(r) l(r)) x fp(r)$ where $fp(r)$ is profit in the final state of r, $p(r)$ is the number of states agent is in profit and $l(r)$ is the number of states agent is not in profit.

Expected utility & optimal agents

Expected utility

P(r|Ag,Env) to be denoted as probability that run r occurs when agent Ag is placed in environment Env.

Expected utility(for run): u(r)P(r|Ag,Env)

Optimal agent $Ag_{opt}$ is the one that maximizes expected utility over all possible runs in the environment

Difficulties with utility-based approaches

  1. Not obvious how to generate the numbers
  2. People don’t think in terms of utilities
  3. Hard to formulate tasks in these terms

Predicate(断言) task specifications

A special case of assigning utilities to runs is to assign 0 (false) or 1 (true). Denote predicate task specification by function Ψ.

Types of tasks

Achievement tasks(at least bring about one of the good states)

Are those of the form achieve state of affairs $\psi$

An achievement task is specified by a set G of “good” or “goal” states: G ⊆ E. The agent succeeds if it is guaranteed to bring about at least one of these states (we do not care which one — they are all considered equally good)

Maintenance tasks(never to bring about one of the bad states)

Are those of the form maintain state of affairs \psi

A maintenance task is specified by a set B of “bad” states: B ⊆ E. The agent succeeds if it is guaranteed never to bring about one of these states, i.e. it manages to avoid all states in B, it never performs actions which result in any state in B occurring
